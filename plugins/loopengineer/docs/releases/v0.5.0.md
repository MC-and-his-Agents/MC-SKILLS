# v0.5.0 Manual Release Plan

Issue: #81

`v0.5.0` is the M6 release checkpoint for LoopEngineer. It records the
runtime-neutral CLI/JSON Loop Engine contract, session preflight admission
reminder, and optional MCP adapter. It does not mean LoopEngineer is a runtime.

## Included Scope

- Runtime-neutral Loop Engine interface contract and ADR.
- `engineContractVersion` compatibility boundary.
- Unified CLI/JSON engine entrypoint at `scripts/loopengineer.py`.
- Session preflight / admission reminder capability.
- Optional stdio MCP adapter over the CLI/JSON engine contract.
- Version, metadata, plugin manifest, and changelog update to `0.5.0`.

## Not Included

- Stable external adapter contract.
- `consume_report` as an engine or MCP capability.
- Lifecycle hooks.
- Worker, scheduler, watcher, automation, gate, merge, closeout, or release
  runtime behavior.
- GitHub, git, CI, PR, release, production host, or `.loom` mutation.
- Package-manager publishing.
- Any release beyond the Git tag and GitHub Release.

## Compatibility

- pluginApiVersion: 1
- protocolVersion: 1
- engineContractVersion: 1
- schemaMajorVersion: 1
- skillContractVersion: 1
- adapterContractVersion: 0

`engineContractVersion: 1` means the CLI/JSON engine contract is stable for M6.
`adapterContractVersion: 0` means the MCP adapter remains experimental and is
not a stable external adapter contract.

## Checklist

- `VERSION` contains `0.5.0`.
- `metadata/loopengineer.json.version` matches `VERSION`.
- `metadata/loopengineer.json.engineContractVersion` is `1`.
- `metadata/loopengineer.json.adapterContractVersion` is `0`.
- `.codex-plugin/plugin.json.version` matches `VERSION`.
- `CHANGELOG.md` has a `0.5.0` entry with compatibility fields.
- Existing `skills/*/skill.yaml` files declare `version` and
  `skillContractVersion`.
- Existing `schemas/v*/...schema.json` files declare `$id`, `schemaVersion`,
  and `kind`.
- Skill entrypoints under `skills/` are present and reference existing files.
- Schema examples under `schemas/v1/examples/` remain valid or intentionally
  invalid according to tests.
- Tests pass with `python3 -m unittest discover -s tests`.
- Diff hygiene passes with `git diff --check`.
- Release readiness passes with `python3 scripts/check_release_readiness.py`.

## Manual Release Rule

After the checklist passes on `main`, dispatch `.github/workflows/manual-release.yml`
with:

```text
release_version: v0.5.0
allow_explicit_commit: false
draft_release: false
```

The workflow creates the tag and non-draft GitHub Release only after release
readiness passes.

## Release Notes Shape

Use this structure for release notes:

```markdown
## Summary
## Scope
## Validation
## Compatibility
## Known Limitations
```

The compatibility section must include:

- pluginApiVersion
- protocolVersion
- engineContractVersion
- schemaMajorVersion
- skillContractVersion
- adapterContractVersion
